## Overview

This guide is designed for AI assistants (agents) helping users migrate from HeroUI v2 to v3. It implements an **incremental migration approach** that analyzes the project first and migrates components in manageable batches rather than all at once.

<Callout type="warning">
  **Critical Constraint**: HeroUI v2 and v3 **cannot coexist** in the same project. The project will be broken during migration. Always work in a feature branch and migrate all component code before switching dependencies.
</Callout>

## Key Principles

1. **Incremental Migration**: Never migrate all components at once. Large projects require phased migration.
2. **Project Analysis First**: Always analyze the codebase to understand component usage before creating a migration plan.
3. **MCP Prompt-Driven**: **Important**: Prompts are registered on the MCP to help with the migrations. Agents should guide users on which prompts to use at each step.
4. **Broken State Management**: **Critical**: v2 and v3 cannot coexist. The project will be broken during migration. Plan for this by:
   - Migrating in a feature branch
   - Preparing all code changes before switching dependencies
   - Having a rollback plan
   - Testing in isolated environment first
5. **All-or-Nothing Dependency Switch**: Once dependencies are updated to v3, ALL components must be migrated. Plan component migrations before switching dependencies.

## Major Changes in v3

- **No Provider Required**: v3 removes the need for `HeroUIProvider`
- **CSS-First Theming**: New CSS-based theming system replaces Tailwind plugin
- **React 19 Required**: v3 requires React 19+ (v2 supports React 18+)
- **No Framer Motion**: Animation library dependency removed
- **Component API Updates**: Many components use React Aria Components patterns
- **Compound Components**: New compound component patterns for better customization
- **Hooks Removed**: v2 component hooks like `useSwitch` or `useInput` are removed - use compound components instead to customize component structure

## Dependency Updates

### Update React to v19+

v3 requires React 19+. This can be done before switching HeroUI (won't break project). Update both `react` and `react-dom`.

### Update Tailwind CSS to v4

v3 requires Tailwind CSS v4 (NOT v3). This can be done before switching HeroUI (won't break project).

### Update HeroUI Packages

- Remove v2 packages: `@heroui/react` and `@heroui/theme`
- Install v3 packages: `@heroui/styles@beta` and `@heroui/react@beta`
- **Critical**: Only do this AFTER all component code is migrated

### Remove Framer Motion

v3 no longer requires Framer Motion. Can be removed when switching to v3 dependencies.

## Configuration Changes

### Remove Tailwind Plugin Configuration

**v2 Configuration:**

```js
// tailwind.config.js
const {heroui} = require("@heroui/react");

module.exports = {
  content: [
    "./src/**/*.{js,ts,jsx,tsx}",
    "./node_modules/@heroui/theme/dist/**/*.{js,ts,jsx,tsx}",
  ],
  plugins: [heroui()],
  // ... other config
};
```

**v3 Configuration:**

Remove the `heroui()` plugin from your Tailwind config. If you only used Tailwind for HeroUI and have no other customizations, you can remove `tailwind.config.js` entirely. Otherwise, keep the file but remove the HeroUI plugin configuration.

### Update CSS Imports

**v2 CSS:**

```css
/* globals.css or main.css */
@tailwind base;
@tailwind components;
@tailwind utilities;
```

**v3 CSS:**

```css
/* globals.css or main.css */
@import "tailwindcss";
@import "@heroui/styles";
```

<Callout type="warning">
  Import order matters! Always import `tailwindcss` before `@heroui/styles`.
</Callout>

### Remove Theme Plugin File

If a `hero.ts` file was created for v2, it can be removed:

```bash
rm hero.ts
```

## Remove HeroUIProvider

v3 does not require a Provider component. Remove it from the application root.

**v2 Code:**

```tsx
// app.tsx or App.tsx
import {HeroUIProvider} from "@heroui/react";

function App() {
  return (
    <HeroUIProvider>
      <YourApplication />
    </HeroUIProvider>
  );
}
```

**v3 Code:**

```tsx
// app.tsx or App.tsx
function App() {
  return <YourApplication />;
}
```

**If Provider props were used:**

If Provider props like `navigate`, `useHref`, `locale`, `disableAnimation`, etc. were used, handle these differently:

- **Router integration**: Use React Router or routing library directly
- **Locale**: Use React Aria's `I18nProvider` directly if needed
- **Animation**: See the Animation Changes section below

### Animation Changes

v3 removes the Framer Motion dependency and handles animations differently:

- **CSS-based animations**: v3 uses native CSS animations and transitions instead of JavaScript-based animations
- **Better performance**: CSS animations provide better performance and smoother animations
- **No global disable**: Unlike v2's Provider `disableAnimation` prop, there's no global animation toggle in v3
- **Per-component control**: Control animations through CSS or component-specific props where available
- **Custom animations**: Use standard CSS `@keyframes` and transition properties for custom animations

## Hooks Removal

HeroUI v2 provided component hooks (like `useSwitch`, `useInput`, `useCheckbox`, etc.) that returned prop getters (`getBaseProps`, `getWrapperProps`, `getThumbProps`, etc.) to customize component structure when users couldn't directly modify inner child components. HeroUI v3 solves this with compound components, eliminating the need for hooks.

**Migration Strategy:**

1. **Identify hook usage**: Search codebase for imports from `@heroui/react` that include hook names (`useSwitch`, `useInput`, `useCheckbox`, `useRadio`, etc.)
2. **Replace with compound components**: Instead of using hooks with prop getters, use the compound component pattern
3. **Reference component guides**: Use `get_component_guides` MCP tool to fetch specific migration guides

**Key Differences:**

- **v2**: Hooks provided prop getters to customize fixed component structures
- **v3**: Compound components allow direct composition of component parts

For detailed migration examples, use the `get_component_guides` MCP tool to fetch component-specific migration guides.

## Component Import Changes

All components are now imported from a single package:

**v2 Imports:**

```tsx
// Individual packages (if used)
import {Button} from "@heroui/button";
import {Card} from "@heroui/card";

// Or from main package
import {Button, Card} from "@heroui/react";
```

**v3 Imports:**

```tsx
// All components from single package
import {Button, Card} from "@heroui/react";
```

### TypeScript Considerations

If using TypeScript, be aware of type changes in v3:

```tsx
// Import types alongside components
import {Button, type ButtonProps} from "@heroui/react";

// Compound component types are properly exported
import {Checkbox, type CheckboxProps} from "@heroui/react";

// Type names may have changed - check component documentation
type MyButtonProps = ButtonProps & {
  customProp?: string;
};
```

**Common type changes:**
- Component prop interfaces may have different names or properties
- Event handler types changed from `onClick` to `onPress` (uses React Aria types)
- Compound component parts have their own type exports
- Ref types updated to match React 19 patterns

## Key API Changes

### General Props Changes

- `key` prop ‚Üí `id` and `textValue` props (ListBox, etc.)
- `onClick` ‚Üí `onPress` (all interactive components)
- `classNames` prop ‚Üí `className` prop (all components)

### Compound Component Patterns

Many components now use compound component patterns for better composition and flexibility. For example:

- **Checkbox**: `Checkbox`, `Checkbox.Control`, `Checkbox.Indicator`, `Checkbox.Content`
- **Card**: `Card`, `Card.Header`, `Card.Title`, `Card.Description`, `Card.Content`, `Card.Footer`
- **Modal**: `Modal`, `Modal.Overlay`, `Modal.Content`, `Modal.Header`, `Modal.Footer`, `Modal.Body`
- **Select**: `Select`, `Select.Trigger`, `Select.Value`, `Select.Content`, `Select.Item`
- **Dropdown**: `Dropdown`, `Dropdown.Trigger`, `Dropdown.Content`, `Dropdown.Item`

See specific component migration guides for detailed examples and usage patterns. Use `get_component_guides` MCP tool to fetch component-specific guides.

## Custom Theme Overrides

If custom theme overrides were used in v2, update them for v3's CSS-based theming system.

**v2 Theme Customization:**

```js
// tailwind.config.js
const {heroui} = require("@heroui/react");

module.exports = {
  plugins: [
    heroui({
      themes: {
        light: {
          colors: {
            primary: {
              // custom colors
            },
          },
        },
      },
    }),
  ],
};
```

**v3 Theme Customization:**

v3 uses CSS variables. Override them in your CSS:

```css
/* globals.css */
@import "tailwindcss";
@import "@heroui/styles";

:root {
  --color-primary: /* your color */;
  /* other CSS variables */
}
```

Check the [Theming documentation](/docs/handbook/theming) for available CSS variables.

## Styling Migration

After component migration is complete, proceed with styling changes. This is a separate step to ensure component functionality is verified before addressing visual changes.

**Styling Migration Guide:**

Use the `get_styling_migration_guide` MCP tool for comprehensive styling migration details:

- Utility class changes (`text-tiny` ‚Üí `text-xs`, `rounded-small` ‚Üí `rounded-sm`, etc.)
- Color token updates (`bg-primary` ‚Üí `bg-accent`, `bg-content1` ‚Üí `bg-surface`, etc.)
- Component styling differences (sizes, spacing, border radius)
- CSS variable changes
- Visual differences and alignment changes

**Key Styling Changes:**

- **Utility Classes**: Custom utilities replaced with standard Tailwind classes
- **Color Tokens**: `primary` ‚Üí `accent`, `secondary` removed, `content1-4` ‚Üí `surface`/`overlay`
- **Numbered Scales**: Color scales like `primary-50`, `primary-100` removed
- **Border Radius**: Default values changed (smaller in v3)
- **Component Styles**: Updated default sizes, padding, and spacing

**Important**: Styling migration happens AFTER component migration and dependency switch.

## Migration MCP Prompts

The Migration MCP server exposes prompts that **users invoke** (not agents). Each prompt includes guidance on which prompts to invoke next, so agents should guide users to follow the prompt workflow.

**Important**: Prompts are invoked by users through their MCP client (e.g., Cursor slash commands, Claude Code prompts menu). You cannot invoke prompts directly - guide users to invoke them.

### Available Prompts

1. **`analyze-and-plan`**: Analyze the project to identify all HeroUI v2 components and create an incremental migration plan. First step in migration. No arguments required.
2. **`implement-migration`**: Implement component migrations incrementally with checkpoints for user approval. Use this to migrate components phase by phase, including the final styling migration phase. No arguments required.

**Agent role**: Guide users to invoke prompts at appropriate stages. Help interpret prompt outputs and apply migrations. Use MCP tools (`get_component_guides`, `get_styling_migration_guide`) to fetch detailed guides when needed.

**Critical**: The `implement-migration` prompt emphasizes stopping at checkpoints for user approval. Always wait for explicit user approval between phases - do not proceed automatically.

## Incremental Migration Workflow

**IMPORTANT**: Since v2 and v3 cannot coexist, the migration happens in two main stages:

1. **Preparation Stage**: Migrate all component code while still on v2 dependencies (code will be broken)
2. **Switch Stage**: Update dependencies to v3 and fix any remaining issues

**‚ö†Ô∏è CRITICAL: Don't build to check for errors during migration**

- Use **typecheck** (e.g., `tsc --noEmit`) if available to check TypeScript errors
- Use **lint** (e.g., `eslint`, `biome check`) if available to check code quality
- **DO NOT** run build commands (e.g., `npm run build`, `next build`, `vite build`)
- **DO NOT** attempt to start/run the project during migration

### Phase 0: Setup and Analysis

1. **Create migration branch**
   - Create a feature branch for migration work
   - Example: `git checkout -b migrate/heroui-v3`

2. **Verify Migration MCP is configured**
   - Check that Migration MCP server is connected
   - Ensure `heroui-react` MCP is NOT connected (to avoid confusion)
   - Verify prompts are available

3. **Guide user to invoke `analyze-and-plan` prompt**
   - This combines analysis and plan creation into one step
   - Help interpret results: component inventory, usage patterns, dependencies, migration plan
   - Review the plan when user shares it

### Phase 1: Dependency Preparation (Before Code Changes)

These steps can be done before switching HeroUI dependencies and won't break the project:

1. **Update React to v19** (if not already)
   - This can be done before switching HeroUI

2. **Update Tailwind CSS to v4** (if not already)
   - This can be done before switching HeroUI

3. **üõë CHECKPOINT: Stop and wait for user approval**
   - **DO NOT proceed to next phase automatically**
   - Explain configuration changes
   - Wait for explicit user approval before continuing

### Phase 2-N: Code Migration (v2 Dependencies Still Active)

**Critical**: During this phase, code will reference v3 APIs but v2 dependencies are still installed. The project will be broken. This is expected and normal.

For each component group in the migration plan:

1. **Guide user to invoke `implement-migration` prompt**
   - Prompt includes instructions for incremental migration with checkpoints
   - **Critical**: Prompt emphasizes stopping between phases for user approval

2. **Fetch component-specific guides**
   - Use `get_component_guides` MCP tool for each component
   - Review API changes, prop migrations, structure changes

3. **Apply code migrations**
   - Migrate component code to v3 API patterns
   - Update imports, props, component structure
   - **Note**: Code will be broken until dependencies are switched

4. **Handle dependencies**
   - If components have dependencies, migrate dependencies first
   - Check if dependencies are already migrated
   - Migrate shared code as needed

5. **üõë CHECKPOINT: Stop and wait for user approval**
   - **DO NOT proceed to next phase automatically**
   - Summarize what was migrated in this phase
   - Wait for explicit user approval before continuing

6. **Document migration status**
   - Track which components have been migrated
   - Note any issues or concerns

### Phase Final: Dependency Switch and Fixes

**Critical**: Only proceed when ALL components have been migrated to v3 API patterns.

1. **Update dependencies**
   - Remove `@heroui/react` and `@heroui/theme` (v2)
   - Install `@heroui/react@beta` and `@heroui/styles@beta` (v3)
   - Remove `framer-motion` if present
   - Update CSS imports (add `@import "@heroui/styles";`)
   - Remove HeroUIProvider from app root
   - Update Tailwind config (remove `heroui()` plugin)

2. **Fix remaining issues**
   - Run typecheck/lint if available (do NOT build)
   - Fix any TypeScript errors reported by typecheck
   - Fix any linting errors
   - Note: Do not attempt to build or run the project during migration

3. **üõë CHECKPOINT: Stop and wait for user approval**
   - **DO NOT proceed to styling automatically**
   - Verify components work correctly
   - Wait for explicit user approval before styling migration

4. **Continue with styling migration** (part of `implement-migration` prompt)
   - The prompt includes instructions for the final styling phase
   - Use `get_styling_migration_guide` MCP tool to fetch styling guide
   - Apply styling updates systematically

5. **Apply styling updates**
   - Use `get_styling_migration_guide` MCP tool
   - Update utility classes, color tokens, CSS variables
   - Test visual appearance

6. **Final verification**
   - Run typecheck/lint one final time (do NOT build)
   - Verify all styling updated correctly
   - Note: Full testing (visual, functionality, accessibility) should be done after migration is complete, not during migration

## Migration Strategies

### Strategy 1: By Dependency (Recommended)

- Migrate foundational components first (Button, Input, Card, etc.)
- Then migrate components that depend on them
- Best for projects with complex component hierarchies
- Ensures dependencies are ready before dependents

**Example order**:
1. Button, Input, Link (foundational)
2. Card, Modal (use Button)
3. Form, Dropdown (use Input, Button)
4. Complex components (use multiple dependencies)

### Strategy 2: By Feature

- Migrate all components in a feature/module together
- Good for feature-based code organization
- Allows feature-by-feature testing
- May require migrating dependencies first

**Example**:
- Feature: User Authentication
  - Migrate: Input, Button, Form, Modal (all auth-related)
- Feature: Dashboard
  - Migrate: Card, Tabs, Select (all dashboard-related)

### Strategy 3: By Frequency

- Migrate most-used components first
- Provides quick wins and early validation
- Good for large codebases with clear usage patterns
- Still need to handle dependencies

**Example**:
1. Button (used 150 times)
2. Input (used 120 times)
3. Card (used 80 times)
4. ... (continue by usage count)

## Best Practices for Agents

1. **Warn about broken state**
   - Always inform user that project will be broken during migration
   - Recommend using a feature branch
   - Set expectations about when project will work again

2. **Migrate all code before switching dependencies**
   - Complete all component code migrations first
   - Only switch dependencies when ALL components are migrated
   - This minimizes the broken state duration

3. **Guide users through prompts**
   - Guide users on which prompt to use at each step
   - Help interpret prompt outputs
   - Use MCP tools to fetch detailed guides
   - **Critical**: Always stop at checkpoints and wait for user approval between phases

4. **Work in a feature branch**
   - Always recommend creating a migration branch
   - Allow user to continue working on main branch
   - Enable easy rollback if needed

5. **Document migration status**
   - Track which components have been migrated
   - Note any issues or concerns
   - Keep a checklist visible
   - Update status after each phase

6. **Handle errors gracefully**
   - If a component migration fails, document why
   - Continue with other components
   - Return to failed components after dependency switch
   - Some issues may resolve once dependencies are updated

7. **Verify after dependency switch**
   - Only after dependencies are updated to v3
   - Run typecheck/lint (do NOT build)
   - Fix type/lint errors as they arise
   - Don't proceed to styling until typecheck/lint passes

## Common Scenarios

### Large Project (100+ components)

- Use smaller batch sizes (3-5 components)
- Prioritize by dependency or frequency
- Allow for multiple sessions
- Create checkpoints between phases
- Use "by-dependency" strategy
- Document progress clearly

### Small Project (\<20 components)

- Can use larger batches (5-10 components)
- May complete in fewer phases
- Still verify incrementally
- Can use any strategy
- Still need to migrate all code before switching deps

### Mixed v2/v3 Usage

- **Not possible**: v2 and v3 cannot coexist
- Must migrate all components before switching dependencies
- Use feature branch to maintain working main branch
- Complete migration in one go

## Troubleshooting

### Issue: MCP prompts not available

- Verify Migration MCP server is connected
- Check server supports prompts capability
- Ensure using correct MCP server (migration, not heroui-react)
- Restart MCP client if needed

### Issue: Migration plan too aggressive

- Reduce batch size
- Use "by-dependency" strategy
- Break phases into smaller steps
- Guide user to re-invoke `create-migration-plan` with adjusted parameters

### Issue: Components failing to migrate

- Check component migration guide using `get_component_guides` tool
- Verify prerequisites are met
- Check if dependencies need to be migrated first
- Document the issue and continue with other components
- Return to failed components after dependency switch (may work once deps are updated)

### Issue: Project broken after code migration

- **Expected**: Project will be broken until dependencies are switched
- Complete all code migrations first
- Only switch dependencies when ready
- Use feature branch to maintain working main branch
- Verify all components migrated before switching

### Issue: Too many errors after dependency switch

- Verify all components were migrated before switch
- Check for missed v2 patterns
- Search for remaining v2 imports and patterns
- Consider rolling back and completing missed migrations
- Fix errors incrementally

## Next Steps

After completing migration:

1. Remove v2 dependencies (already done in dependency switch)
2. Switch from Migration MCP to `heroui-react` MCP for v3 development
3. Update documentation references
4. Run final verification
5. Merge migration branch to main
